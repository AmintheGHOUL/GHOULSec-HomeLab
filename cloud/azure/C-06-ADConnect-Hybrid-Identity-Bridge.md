# C-06 — Hybrid Identity Bridge: AD Connect Sync & MSOL DCSync Rights

**Tactic:** Persistence · Privilege Escalation · Credential Access  
**MITRE:** T1078.002 · T1003.006 · T1484.002  
**Severity:** Critical  
**Platform:** On-premises AD (ghoul.local) + Microsoft Entra ID  
**Tools:** Microsoft Entra Connect, AADInternals, Mimikatz (deferred — evasion phase)

---

## Overview

Microsoft Entra Connect (formerly Azure AD Connect) is the synchronisation engine that bridges on-premises Active Directory with Microsoft Entra ID. It is deployed in the vast majority of enterprise Microsoft environments — any organisation using both on-prem AD and Microsoft 365 or Azure has some form of hybrid identity.

The attack surface it creates is significant and frequently overlooked. When AD Connect is installed, it automatically creates a highly privileged service account (`MSOL_xxxxxxxx`) in Active Directory with **DS-Replication** rights — the same rights required to perform a DCSync attack. If an attacker can extract this account's credentials from Azure, they can DCSync the entire domain without ever touching a domain controller directly.

This is the technique used in the **SolarWinds supply chain compromise** — attackers pivoted from a compromised cloud identity to on-prem AD by abusing the AD Connect sync account, allowing them to move laterally across thousands of victim networks.

In GHOULSec, this scenario demonstrates the full hybrid kill chain: a cloud-side attack (password spray → Entra compromise) becomes a complete on-prem domain compromise via the sync account.

## Phase 1 — AD Connect Installation

### Prerequisites

AD Connect requires TLS 1.2 on Windows Server. On a fresh Windows Server 2019 DC this is often not enabled by default. The following registry keys must be set before installation:

```powershell
# Enable TLS 1.2 for AD Connect
New-Item 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Internet Settings\WinHttp' -Force
Set-ItemProperty -Path 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Internet Settings\WinHttp' -Name 'DefaultSecureProtocols' -Value '0x00000800' -Type DWord

New-Item 'HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.2\Server' -Force
Set-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.2\Server' -Name 'Enabled' -Value '1' -Type DWord
Set-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\TLS 1.2\Server' -Name 'DisabledByDefault' -Value '0' -Type DWord

New-Item 'HKLM:\SOFTWARE\Microsoft\.NETFramework\v4.0.30319' -Force
Set-ItemProperty -Path 'HKLM:\SOFTWARE\Microsoft\.NETFramework\v4.0.30319' -Name 'SchUseStrongCrypto' -Value '1' -Type DWord
```

DC01 internet connectivity confirmed before installation:

```powershell
Test-NetConnection -ComputerName login.microsoftonline.com -Port 443
```

## Phase 4 — DCSync Rights Confirmation

The most critical finding — verifying that `MSOL_<auto-generated-id>` has DS-Replication rights on the domain root:

```powershell
$acl = Get-ACL "AD:\DC=ghoul,DC=local"
$acl.Access | Where-Object {
    $_.IdentityReference -like "*MSOL*" -and 
    $_.ActiveDirectoryRights -like "*ExtendedRight*"
} | Select IdentityReference, ActiveDirectoryRights, ObjectType
```

**Output:**

```
IdentityReference       ActiveDirectoryRights ObjectType
-----------------       --------------------- ----------
GHOUL\MSOL_<auto-generated-id> ExtendedRight 1131f6aa-9c07-11d1-f79f-00c04fc2dcd2
GHOUL\MSOL_<auto-generated-id> ExtendedRight 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2
GHOUL\MSOL_<auto-generated-id> ExtendedRight 00299570-246d-11d0-a768-00aa006e0529
```

> **Screenshot 4:** PowerShell output showing MSOL_<auto-generated-id> with both DCSync GUIDs confirmed

**GUID mapping — what these rights mean:**

| GUID | Right | Impact |
|---|---|---|
| `1131f6aa-9c07-11d1-f79f-00c04fc2dcd2` | DS-Replication-Get-Changes | Required for DCSync |
| `1131f6ad-9c07-11d1-f79f-00c04fc2dcd2` | DS-Replication-Get-Changes-All | Required for DCSync (includes secrets) |
| `00299570-246d-11d0-a768-00aa006e0529` | User-Force-Change-Password | Can reset any user's password |

Both DCSync GUIDs are present. **`MSOL_<auto-generated-id>` can dump every password hash in `ghoul.local`.**

## Phase 5 — The Hybrid Kill Chain

With the sync account identified and DCSync rights confirmed, the full attack chain from cloud to on-prem is:

```
[Kali] Password spray → jsmith compromised (C-05)
            │
            ▼
[ROADrecon] Directory dump → Sync_GHOUL-DC0... account identified in Entra
            │
            ▼
[AADInternals] Get-AADIntSyncCredentials → MSOL_<auto-generated-id> plaintext password
            │
            ▼  ← Cloud-to-on-prem pivot
[Mimikatz/Impacket] DCSync ghoul.local using MSOL credentials
            │
            ▼
[KRBTGT hash extracted] → Golden Ticket forged
            │
            ▼
[ghoul.local fully compromised] — persistent access survives password resets
```

### Credential Extraction — Tooling Note

The credential extraction step using `Get-AADIntSyncCredentials` (AADInternals) was attempted but failed due to Microsoft retiring the `provisioningapi.microsoftonline.com` endpoint in 2024. This API was the backend for AADInternals credential extraction in versions ≤0.8.x.

**This is a real-world finding worth documenting:**

> Microsoft patched this specific extraction vector after it was used in the SolarWinds compromise. The `provisioningapi.microsoftonline.com` endpoint was deprecated and the credential storage moved to a more secure architecture. This demonstrates that Microsoft does respond to published attack techniques — though the underlying risk (MSOL account with DCSync rights) remains.

**Alternative extraction paths that still work:**

Local extraction via ADSync database — the credentials are stored encrypted in `C:\Program Files\Microsoft Azure AD Sync\Data\ADSync2019\ADSync.mdf` using the ADSync service key. Decryption requires either the service key or running as SYSTEM on the AD Connect server.

The DCSync attack itself is deferred to the evasion techniques phase — executing Mimikatz with AMSI bypass and obfuscation for a realistic attack demonstration. Windows Defender detected the standard Mimikatz binary immediately:

> **Screenshot 5 (from Defender alerts):** `HackTool:Win32/Mimikatz` quarantined three times — High severity

This detection is intentional and demonstrates why real attackers use obfuscated or reflectively-loaded variants. The evasion phase will cover AMSI bypass, Invoke-Mimikatz, and in-memory execution.

```xml
  </mitre>

```

### Sentinel KQL — DCSync Detection via Replication Rights

```kql
SecurityEvent
| where EventID == 4662
| where Properties has "1131f6aa-9c07-11d1-f79f-00c04fc2dcd2"
   or Properties has "1131f6ad-9c07-11d1-f79f-00c04fc2dcd2"
| where SubjectUserName !has "MSOL_"
| where SubjectUserName !has "GHOUL-DC01$"
| project TimeGenerated, SubjectUserName, SubjectDomainName, Properties
// Alert: DCSync performed by non-MSOL, non-DC account = likely attacker
```

### Sentinel KQL — AD Connect Sync Account Used From Unexpected IP

```kql
SigninLogs
| where TimeGenerated > ago(1h)
| where UserPrincipalName has "Sync_GHOUL"
| where IPAddress !startswith "192.168.10."
| project TimeGenerated, UserPrincipalName, IPAddress, AppDisplayName, ResultType
// Sync account should only authenticate from DC01 — any other IP is suspicious
```

## Remediation

**Fix 1 — Tier the AD Connect server**

AD Connect should run on a dedicated Tier 0 server, not on the Domain Controller. Running it on DC01 concentrates risk — if the server is compromised, both the sync credentials and domain secrets are accessible.

**Fix 2 — Monitor MSOL account activity**

The MSOL account should never:
- Authenticate interactively
- Authenticate from any IP other than the AD Connect server
- Be used for any purpose other than directory synchronisation

Create a Sentinel alert for any MSOL account activity outside normal sync operations.

**Fix 3 — Use Managed Service Accounts**

Newer AD Connect versions use Group Managed Service Accounts (gMSA) instead of the MSOL account. The `ADSyncMSA<id>$` account seen in GHOULSec is a gMSA — its password is automatically managed and cannot be extracted via the old AADInternals method. Verify this is configured correctly.

**Fix 4 — Enable Entra ID Protection**

Entra ID Protection detects anomalous sign-in patterns for synced accounts including the Sync account. Enable it and configure risk-based Conditional Access policies.

**Fix 5 — Restrict who can read AD Connect configuration**

The ADSync database and configuration should only be readable by the ADSync service account and Domain Admins. Audit permissions on `C:\Program Files\Microsoft Azure AD Sync\`.

## Real-World Reference — SolarWinds

The SolarWinds supply chain compromise (2020) used this exact technique at scale. After gaining initial access via the trojanised Orion update, NOBELIUM (APT29/Cozy Bear) used the AD Connect sync account to pivot from cloud identities to on-premises Active Directory across thousands of victim networks.

Microsoft's post-incident analysis specifically called out the hybrid identity attack path:

> *"The actor used the high privileged account in the on-premises environment to create credentials for the cloud service principal... this allowed the actors to maintain persistence in the cloud environment even if the on-premises foothold was remediated."*

The GHOULSec hybrid bridge demonstrates exactly this attack path at lab scale — with the detection controls that would have caught it.
